🟦 Day3：データベースとは何か（概念＋実作業）

今日は アプリの“土台”となるデータベース（DB）を理解し、実際に作る日 です。
まずは「データをどこに保存するのか」という視点で整理していきます。

■ まず超シンプルに説明すると…

データベース＝アプリのデータをしまう “大きな箱”。

（ユーザー情報、投稿、商品データなど、全部ここに入る）

ただし本質はもう少し深いです。



■ 本質的に大切な考え方

✔ 1) データベースは「アプリが動くための土台」

PHP や Laravel がデータを保存・取得するには、
まず “入れ物（データベース）” が絶対に必要です。

この箱がないと、アプリは何も保存できません。



✔ 2) MySQL は箱（データベース）の中に「表（テーブル）」を置く仕組み

データベース＝大きな箱

テーブル＝箱の中の「Excelのシート」

例：usersテーブルには名前・メールなどが並ぶ

Day3 はまだ「箱」を作っただけで、
中に入れるテーブルは Day4〜で作っていきます。



✔ 3) Docker の設定で「最初から箱が作られる」ことがある

Day2 の docker-compose.yml に

MYSQL_DATABASE: app_db

と書いたため、MySQL 起動時に 自動で app_db が作成されていた。

つまり、

MySQL が勝手に作ったのではない

Day2 の設定により “正しく自動生成された”

これは実務でもよく使われる仕組み。



■ 今日の実作業

✔ Docker コンテナ名（mysql_local）を確認
✔ MySQL にログイン

docker exec -it mysql_local mysql -u root -p

✔ CREATE DATABASE を試すと「すでに存在」と表示
→ Docker の設定により事前に作成されていたため

✔ SHOW DATABASES; で app_db が存在することを確認
→ Day3 のゴール達成



■ 今日学んだ疑問の答え

✔ SQL の大文字・小文字は意味に影響しない

CREATE / create / Create → すべて同じ

ただしプロは、
キーワード＝大文字, 名前＝小文字
で読みやすい書き方をする



✔ データベースは作り直せる
DROP DATABASE app_db;
CREATE DATABASE app_db;

の2ステップだけでOK
（ただし DROP は完全削除なので注意）



✔ app_db が最初からあった理由

ChatGPTが勝手に作ったわけではない

docker-compose.yml の設定が正しく働いただけ
→ これは実務でもよくある流れ



🟩 今日のまとめ（本質）

データベースは「アプリのすべてのデータをしまう土台」。
Day3 ではその土台となる箱を確認し、
MySQL・Docker の動きを理解することができた。

これで Day4（テーブル作成） に進む準備が完全に整いました。
